# Skill 28: Identity-Aware Row-Level Security for AI / MCP Agents

**Category:** Security | MCP | RLS | **Level:** Advanced

---

## Overview

When an AI agent talks to your Oracle database through an MCP (Model Context Protocol) server, every NL2SQL question, every report, and every tool call usually flows through one shared database connection — no matter which human (or agent acting on their behalf) actually asked the question. Without extra work, the database can't tell "Alice from Finance" apart from "Bob the contractor" or "an over-eager agent with SQL access." Access rules end up living only in application code, and anything that bypasses the app — a reporting tool, an ETL job, a different agent — bypasses those rules too.

Modern OAuth-based MCP servers solve half the problem for you: they propagate the *authenticated caller's identity* into the database session automatically, with nothing for you to build. This skill shows an AI agent (any provider, any model) how to:

1. Discover **who** is actually calling, on a per-request basis, using `SYS_CONTEXT`.
2. Use that identity inside an Oracle **Virtual Private Database (VPD)** / Row-Level Security policy, so the database — not the agent, not the app — enforces who can see or change which rows.
3. Verify the policy actually changes behavior for different identities.
4. Adapt the pattern for MCP gateways or app tiers that don't natively propagate OAuth identity the way Oracle's OCI Database Tools MCP Server does.

The result: a SELECT typed by a human, a report run by an agent, and an ad-hoc NL2SQL query generated by an LLM all get filtered by the *same* database-enforced rule — because the rule lives on the table, not in any one client.

## How It Works

```
End user (Azure Entra ID / AD / IAM identity)
     │
     │  1. Authenticates via OAuth2 (or is acted on by an agent OBO)
     ▼
AI Agent / LLM client  ──tool call──▶  MCP Server (e.g. OCI Database Tools)
     │                                        │
     │                                        │ 2. Validates token, opens DB session,
     │                                        │    sets identity into session context
     ▼                                        ▼
                                    Oracle Database session
                                    ┌─────────────────────────────┐
                                    │ SYS_CONTEXT('USERENV', ...)  │  ← who the DB session/user is
                                    │ SYS_CONTEXT('CLIENTCONTEXT', │  ← who the *end user/agent* is,
                                    │   'OAUTH_SUB', ...)          │     populated per request
                                    └─────────────────────────────┘
                                                │
                                                │ 3. VPD policy function reads the
                                                │    identity and returns a predicate
                                                ▼
                                    Table-level Row-Level Security (VPD)
                                    Oracle silently rewrites every SELECT /
                                    UPDATE / DELETE to add a WHERE clause —
                                    before the optimizer even sees it.
                                                │
                                                ▼
                                  Caller sees only the rows they're entitled to,
                                  no matter how the query arrived (ad-hoc SQL,
                                  NL2SQL, a custom Tool, or a Report).
```

Identity in the session + VPD = data protection that doesn't care how someone got to the table.

## Step-by-Step

### Step 1 — Discover the caller's identity

Most Oracle Database MCP servers that use OAuth2/OIDC populate two namespaces you can query with `SYS_CONTEXT` from any SQL session:

- `USERENV` — standard Oracle session attributes (always available).
- `CLIENTCONTEXT` — populated per-request by the MCP layer from the OAuth token (available specifically when the MCP server propagates end-user identity, e.g. OCI Database Tools MCP Server).

Create a read-only diagnostic Tool (or just run this SQL) so any agent can introspect "who am I, really?" before trusting a session:

```sql
SELECT
  SYS_CONTEXT('USERENV', 'CURRENT_SCHEMA')              AS current_schema,
  SYS_CONTEXT('USERENV', 'SESSION_USER')                AS session_user,
  SYS_CONTEXT('USERENV', 'PROXY_USER')                  AS proxy_user,
  SYS_CONTEXT('USERENV', 'AUTHENTICATED_IDENTITY')      AS authenticated_identity,
  SYS_CONTEXT('USERENV', 'AUTHENTICATION_METHOD')       AS authentication_method,
  SYS_CONTEXT('USERENV', 'ENTERPRISE_IDENTITY')         AS enterprise_identity,
  SYS_CONTEXT('CLIENTCONTEXT', 'OAUTH_SUB')             AS oauth_sub,          -- the authenticated human/agent
  SYS_CONTEXT('CLIENTCONTEXT', 'OAUTH_SUB_TYPE')        AS oauth_sub_type,
  SYS_CONTEXT('CLIENTCONTEXT', 'OAUTH_USER_OCID')       AS oauth_user_ocid,
  SYS_CONTEXT('CLIENTCONTEXT', 'OAUTH_CLIENT_OCID')     AS oauth_client_ocid,
  SYS_CONTEXT('CLIENTCONTEXT', 'OAUTH_CLIENT_NAME')     AS oauth_client_name,
  SYS_CONTEXT('CLIENTCONTEXT', 'OAUTH_DOMAIN_NAME')     AS oauth_domain_name,
  SYS_CONTEXT('CLIENTCONTEXT', 'IAM_DOMAIN_APP_ROLES')  AS iam_domain_app_roles,
  SYS_CONTEXT('CLIENTCONTEXT', 'RESOURCE_OCID')         AS resource_ocid
FROM dual;
```

**What an agent should do with this:** treat `OAUTH_SUB` (or your gateway's equivalent client-context value) as the *true caller identity* for authorization decisions — not the shared database/service account, and not anything the LLM itself claims about who it's acting for. No logon trigger, no custom context package, and no `DBMS_SESSION.SET_CONTEXT` call is required when the MCP layer already populates this for you.

### Step 2 — Decide the access rule

Express the business rule as plain conditions before writing any PL/SQL, e.g.:

> "Internal staff (corporate domain identities) can see every row. Everyone else — external identities, partner agents, or callers with no identity at all — can only see rows that meet an explicit opt-in/eligibility condition."

Write this rule down first; it becomes the `IF` logic in your policy function.

### Step 3 — Write the VPD policy function

A VPD policy function takes a schema/object name and returns a `WHERE`-clause fragment (or `NULL` for "no restriction, see everything"). Template:

```sql
CREATE OR REPLACE FUNCTION app_schema.row_visibility_policy (
  p_schema IN VARCHAR2,
  p_object IN VARCHAR2
) RETURN VARCHAR2 AS
  v_identity VARCHAR2(4000) :=
    SYS_CONTEXT('CLIENTCONTEXT', 'OAUTH_SUB');   -- swap in your identity source
BEGIN
  -- Trusted / internal identities see everything.
  IF v_identity IS NOT NULL
     AND LOWER(v_identity) LIKE '%@your-trusted-domain.com' THEN
    RETURN NULL;                                  -- NULL predicate => unrestricted
  END IF;

  -- Everyone else (including NULL/absent identity): restrict to eligible rows.
  -- Example: only rows where a JSON preference flag is explicitly true.
  RETURN q'[JSON_VALUE(contact_preferences, '$.sms') = 'true']';
END row_visibility_policy;
/
```

Keep the function cheap (a context lookup plus a comparison) — it runs on every qualifying statement.

### Step 4 — Attach the policy with `DBMS_RLS`

```sql
BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema   => 'APP_SCHEMA',
    object_name     => 'YOUR_TABLE',
    policy_name     => 'YOUR_TABLE_VISIBILITY',
    function_schema => 'APP_SCHEMA',
    policy_function => 'ROW_VISIBILITY_POLICY',
    statement_types => 'SELECT, UPDATE, DELETE',  -- restrict writes too, not just reads
    update_check    => TRUE,                       -- block edits that move a row OUT of visibility
    policy_type     => DBMS_RLS.DYNAMIC,           -- re-evaluated every query; safe for pooled/shared sessions
    enable          => TRUE
  );
END;
/
```

Key parameter notes for an agent implementing this:

- `statement_types => 'SELECT, UPDATE, DELETE'` — an identity that can't *see* a row shouldn't be able to update or delete it either.
- `update_check => TRUE` — stops a caller from editing a row they can currently see in a way that pushes it out of their own future visibility.
- `policy_type => DBMS_RLS.DYNAMIC` — on pooled/shared MCP database connections, always prefer `DYNAMIC` over a cached/static policy. Correctness beats a microsecond of performance when the same physical session may serve different identities across requests.

### Step 5 — Test as multiple identities

Before declaring success, an agent (or the human reviewing its work) should verify the policy actually changes results:

1. Run the query/report/tool **as a trusted identity** → expect the unrestricted row count.
2. Run the **same** query/report/tool **as a different (non-trusted) identity**, through the same MCP server, against the same table → expect only the eligible subset.
3. Confirm the *generated SQL* (visible via EXPLAIN PLAN, SQL Monitor, or the database audit trail) actually contains the injected predicate — don't just trust the row count.

```sql
-- Sanity check: this is what an unrestricted caller's query becomes
-- for a restricted caller, once VPD has rewritten it:
SELECT * FROM your_table
 WHERE <policy_predicate_returned_by_your_function>;
```

4. Spot-check via `unified_audit_trail` or your MCP server's logs that the identity attribute you keyed off (`OAUTH_SUB` or equivalent) matches the caller you expect for that session.

### Step 6 — Adapting beyond OCI Database Tools MCP Server

The `CLIENTCONTEXT.OAUTH_*` attributes are specific to Oracle's OCI Database Tools MCP Server. If you're integrating a different AI provider, a self-hosted MCP gateway, or an application-tier connection pool that doesn't propagate OAuth claims into `CLIENTCONTEXT` automatically, reproduce the same pattern manually:

- Have the gateway/app call `DBMS_SESSION.SET_IDENTIFIER` or set a custom application context (via `DBMS_SESSION.SET_CONTEXT` against a package-based context) immediately after authenticating the end user and before running any agent-issued SQL.
- Pass the authenticated subject (user ID, email, or OCID-equivalent) as that identifier/context value, exactly as you would the `OAUTH_SUB` value above.
- Point your VPD policy function at whichever `SYS_CONTEXT` namespace/attribute you used instead of `CLIENTCONTEXT.OAUTH_SUB`.
- Everything downstream (Steps 2–5) stays identical — VPD doesn't care how the identity got into the session, only that it's there and trustworthy.
- If your gateway has no concept of per-end-user identity at all (one shared service account for every caller), this pattern can't help you yet — fix identity propagation first; row-level security is only as good as the identity it's keyed on.

## Security Checklist

- [ ] Confirm your MCP server/gateway actually propagates per-end-user identity into the database session — verify with the Step 1 query before building any policy on top of it.
- [ ] Never authorize based on a claim the LLM/agent makes in natural language — only on values read from `SYS_CONTEXT` (or your gateway's verified equivalent).
- [ ] Default-deny: structure policy functions so an absent/NULL identity gets the *most restrictive* predicate, not the most permissive one.
- [ ] Apply `statement_types` covering writes, not just `SELECT`, wherever a table is mutable.
- [ ] Use `update_check => TRUE` on any policy where visibility could otherwise be gamed via an edit.
- [ ] Prefer `DBMS_RLS.DYNAMIC` for any table reachable through pooled/shared MCP or app-tier connections.
- [ ] Periodically audit `unified_audit_trail` (or equivalent) for the identity attribute you key policies on, to catch unexpected NULLs, spoofing attempts, or misconfigured gateways.
- [ ] Test policies as more than one identity class (trusted + untrusted) before shipping — a policy that's never been observed to actually filter anything hasn't been tested.
- [ ] Re-test after any MCP server, gateway, or IAM domain configuration change — identity propagation is part of your security boundary and can silently break.
